## Task 2: ADR-004 Gap 1 — Deprecate `strategy="auto"` when `enabled=True`

### 1. Open questions

None. ADR-004 §Decision explicitly prohibits automatic backend selection. The deprecation path is clear.

### 2. Goal

Emit a `deprecate()` call (via `utils/deprecations.py`) when a caller sets `enabled=True` without an explicit strategy (or with `strategy="auto"`). This gives users the two-minor deprecation window required by ADR-011 before v1.0.0 removes `strategy="auto"` entirely.

### 3. Why this task exists

ADR-004 §Decision: "no automatic strategy selection." `ParallelConfig(strategy="auto")` is the current default when `enabled=True`, and the `auto_strategy()` heuristic selects a backend based on CPU count, CI detection, OS, and task size — making behavior non-deterministic across machines. The deprecation must land in v0.11.4 (or earlier) so that v1.0.0 can enforce explicit strategy without violating the two-minor window.

### 4. Governing ADRs

- `docs/improvement/adrs/ADR-004-parallel-execution-framework.md`
- `docs/improvement/adrs/ADR-011-deprecation-and-migration-policy.md` (§2 two-minor window)

### 5. Hardening / red-team findings

Appendix Gap 1 (severity 9): `ParallelConfig(strategy="auto")` at `parallel/parallel.py:517-589`. The `auto_strategy()` function is called when `strategy` resolves to `"auto"`. The deprecation must fire when `enabled=True AND strategy == "auto"` — not when `enabled=False` (no backend is selected at all).

### 6. Compatibility posture

Deprecation only; no removal. Code that does not set `enabled=True` is unaffected. Code that sets `enabled=True` with an explicit strategy (not `"auto"`) is unaffected. Only code that relies on automatic backend selection is warned.

### 7. Non-goals

- Do not remove `strategy="auto"` or `auto_strategy()` in this task.
- Do not change the `"auto"` default in `ParallelConfig`; deprecation is sufficient.
- Do not add a CI gate enforcing explicit strategy (that is a v1.0.0-rc obligation).

## Task 5: ADR-038 Gaps 2–3 — Plugin §5 taxonomy enforcement and allowlist comment

### 1. Open questions

**Scope question for Gap 2:** ADR-038 §5 requires third-party plugin config surfaces to follow the `*Spec`/`*Options`/`*Config` taxonomy. But plugin config surfaces are plugin-private (they're validated by `validate_plugin_config_schema`, not inspectable at registration time). What exactly should `validate_plugin_meta` enforce? Options:
- Check that any `*Config`-named metadata key uses the taxonomy (probably not applicable).
- Require plugins with `config_schema` to declare a `config_taxonomy_version` field.
- Add a documentation-level requirement only (close as "enforcement via docs" rather than code gate).

Gap 2 severity is 4 (violation impact 2, scope 2). Given the difficulty of runtime enforcement for plugin-private naming conventions, the most practical closure is: add a documentation requirement in the plugin contribution guide and a comment in `validate_plugin_meta` calling out the ADR-038 §5 expectation. ## Task 7 (CRITICAL): Nightly parity-reference CI mismatch — confirmed root cause: unpinned scikit-learn ≥1.8 changes `DecisionTreeRegressor` split selection

### 1. Open questions — RESOLVED

Root cause is **confirmed** (see Finding below), and the fix-policy question is **decided and implemented**:

- **Pin-back vs. regenerate vs. scope-to-CI-only — decided: scope to CI only.** Two candidates were considered: (a) re-add the upper bound to the project-wide `constraints.txt`, or (b) regenerate the fixtures against scikit-learn≥1.8. Both were rejected in favor of a third option: a `tests/parity_reference/constraints.txt` overlay applied only in the `parity-reference` CI job. Rationale: (b) alone is not durable — bisection showed scikit-learn 1.8.0 and 1.9.0 *already disagree with each other* (3,361 vs 58,523 diffs against the same fixtures), so "regenerate against whatever's current" would only fix today's nightly run and could break again on the next sklearn release with nothing to catch it; the parity harness's entire purpose requires a fixed baseline. Global pin-back (a) would have undone the general-compatibility floor widening that `9693c3d8` deliberately introduced for every consumer, to serve one nightly test's narrower reproducibility need — those are different concerns (general compatibility vs. byte-for-byte determinism) and shouldn't share one constraint file. The scoped overlay gets the stability benefit of (a) without re-narrowing the project-wide floor.
- **Process gap — addressed via documentation, not a new CI gate (yet).** The parity harness is not part of `pytest -q` (it is nightly-only), so the original dependency-unpinning commit's validation never exercised it. `tests/parity_reference/README.md` (new) now documents this sensitivity and the update procedure, so a future intentional sklearn bump has a clear, discoverable path. Adding a PR-time gate for `requirements.txt`/`constraints.txt` changes is left as a smaller follow-up (see Non-goals) rather than bundled into this fix.

### 2. Goal

Restore nightly parity-reference CI to green by re-establishing a deterministic scikit-learn version for `DecisionTreeRegressor`-based explanation paths (regression and probabilistic-regression modes), scoped so it does not narrow the project-wide compatibility floor, and document the rationale so a future contributor doesn't strip the pin without understanding why it exists.

### 3. Why this task exists

The parity-reference harness (`tests/parity_reference/run_parity_reference.py`) is the project's behavioral-stability contract: it asserts that `CalibratedExplainer` output for a fixed seed/model/dataset is identical (within `rtol=1e-6, atol=1e-8`) to committed golden fixtures. Confirmed investigation (below) shows this contract is currently broken for regression-mode explanations because of an uncontrolled upstream dependency upgrade, not a defect in `calibrated_explanations` itself — but it must still be fixed, because every other v0.11.4 task's validation relies on the parity harness staying trustworthy.

### 4. Governing ADRs / standards

- `docs/improvement/adrs/ADR-004-parallel-execution-framework.md` — ruled out as a contributing factor for this specific failure (see Finding); not directly governing the fix, but cross-referenced because `strategy="auto"` was the planning session's initial hypothesis and is now eliminated.
- Dependency-constraint governance (`constraints.txt` "Update policy" header comment) — this task is squarely about that policy: "Add pins ONLY when CI reveals unavoidable incompatibilities" was satisfied retroactively by this investigation; the original removal in the commit below should not have shipped without parity-harness coverage.
- Release-plan validation gate: nightly parity harness is the canonical regression guard for explanation output stability; this task restores that guard's integrity for the rest of v0.11.4.

### 5. Hardening / red-team findings — CONFIRMED root cause (bisected 2026-06-16)

**Introducing commit identified:** `9693c3d8` ("feat: remove unnecessary constraints", 2026-06-12) replaced the exact pins `numpy>=1.24,<2`/`scikit-learn==1.6.1` (Python <3.13) and `numpy>=2.1.2`/`scikit-learn==1.5.2` (Python ≥3.13) in `constraints.txt` with the bare floors already present in `requirements.txt` (`numpy>=1.24`, `scikit-learn>=1.3`), removing all upper bounds. This matches the user's reported timing window (June 12–13) exactly — it is the only commit in that window touching installed dependency versions.

**Local repro on `main` @ `a11e91d8` for `--dataset classification` did NOT fail** (confirmed in the planning session and again now) — this is why the initial investigation pass treated the report as unconfirmed. The actual failure is dataset-specific:

| Dataset | Result (current `main`, fresh nightly-equivalent install) |
| --- | --- |
| `classification` | **PASS** |
| `multiclass` | **PASS** |
| `regression` | **FAIL** — 58,523 diffs (`rule_weight.low/high/predict`, `rule_prediction.*`, rule thresholds) |
| `probabilistic_regression` | **FAIL** — 76,893 diffs (same shape; also `rule.*` text and `value_str`) |

Both `--condition-source observed` and `--condition-source prediction` fail identically for the two regression-mode datasets. The user's quoted command (`--dataset classification`) was almost certainly the nightly job's first sub-step shown in the log; the actual failing sub-steps are the `regression` and `probabilistic_regression` invocations later in the same job (`ci-nightly.yml:36-47` runs all four datasets sequentially under one step).

**Isolated to scikit-learn, not numpy, via dependency-only bisection** (same code at the pre-window commit `37654af`, only installed package versions varied):

| Environment | sklearn | numpy | Result |
| --- | --- | --- | --- |
| Old pins (pre-`9693c3d8`) | 1.6.1 | 1.26.4 | PASS |
| New floors, fresh install (post-`9693c3d8`, current) | 1.9.0 | 2.4.6 | **FAIL** |
| Old sklearn + new numpy | 1.6.1 | 2.4.6 | PASS |
| New sklearn + old numpy | 1.9.0 | 1.26.4 | **FAIL** |

→ numpy 2.x is not implicated. The divergence is entirely attributable to the scikit-learn version.

**Narrowed to the 1.7.2 → 1.8.0 boundary** by testing intermediate scikit-learn releases (code held constant at the pre-window commit, numpy held at `<2`):

| scikit-learn | Result | Diff count (`--dataset regression`) |
| --- | --- | --- |
| 1.6.1 | PASS | 0 |
| 1.7.0 | PASS | 0 |
| 1.7.1 | PASS | 0 |
| 1.7.2 | PASS | 0 |
| 1.8.0 | **FAIL** | 3,361 |
| 1.9.0 | **FAIL** | 58,523 |

This means scikit-learn changed some internal `DecisionTreeRegressor` behavior (most likely splitter tie-breaking or threshold-selection arithmetic — the exact upstream change was not pinpointed in the changelog during this investigation, only its effect) between 1.7.2 and 1.8.0, such that `DecisionTreeRegressor(random_state=42).fit(...)` no longer selects the same split points for this dataset, even though the fit call and inputs are byte-identical. `DecisionTreeClassifier` is unaffected (classification/multiclass both pass at every tested sklearn version up to 1.9.0), which is consistent with the change being in the regressor's continuous-threshold splitter path rather than shared tree-building code.

**Why this slipped through review:** the `9693c3d8` commit message claims "Verified ranges by local + CI testing: 1.5.2 ... 1.7.2, 1.8.0, 1.9.0" — i.e. 1.8.0 *was* tested, but evidently only against `pytest -q` (which does not include the parity-reference harness; it is a standalone script wired only into `ci-nightly.yml`). The verification step never ran the regression-mode parity check, so the divergence was invisible at review time and only surfaced on the next nightly run.
