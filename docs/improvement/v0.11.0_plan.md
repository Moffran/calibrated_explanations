## Global rules (deterministic reproducibility + security)

These rules apply across all v0.11.0 work items (domain model, preprocessing,
serialization, persistence, plugin telemetry).

1. **Canonical hashing of manifests.**
         canonicalized deterministically and hashed. Canonicalization rules
         are strict and MUST be followed verbatim when producing the bytes to
         hash:

         - Numbers must be rendered in shortest decimal form (no gratuitous
           exponential notation) where feasible.
         - Datetimes must be serialized as ISO-8601 UTC (`YYYY-MM-DDThh:mm:ssZ`).
         - `null` must use JSON `null`.
         - Canonical JSON MUST be produced by:

           ```py
           canonical = json.dumps(
                 manifest,
                 sort_keys=True,
                 separators=(",",":"),
                 ensure_ascii=False,
           ).encode("utf-8")
           parameters_hash = "sha256:" + sha256(canonical).hexdigest()
           ```

      - Manifests MUST NOT include raw training data or PII. If sensitive keys
         - A configurable denylist and allowlist of payload keys (strings or
         - Provide an explicit redaction helper that replaces sensitive values
           with the literal string `'<redacted>'` before persistence when
           redaction mode is requested. Record `redaction_applied: true` in the
           manifest when redaction occurred.
         - Provide an allowlist override API for known-safe fields; overrides
           must be auditable and recorded in the manifest metadata.
      - Persist the canonical manifest as an immutable artifact and, where
         permitted, reference it from `metadata.audit.config_artifact` (or
         `metadata.audit.run_id` when artifact storage is unavailable). The
         manifest MUST include `access_control` and `retention_policy` fields
         (see artifact manifest schema below) and documentation must explain
         how artifact storage enforces RBAC and how to revoke access.

      - Minimize runtime and build dependencies: prefer Python stdlib code for
         canonicalization, hashing, and validation. Introduce third-party
         libraries only when the functionality cannot be implemented safely and
         deterministically with the stdlib.

2. **Artifact manifest schema.**
      - All persisted artifacts (parameter manifest, calibrator primitives,
         mappings, saved state) MUST include:
         - `schema_version`
         - `access_control` (e.g., `{"read": ["role:maintainer"], "write": ["role:ci"]}`)
         - `retention_policy` (e.g., `{"class": "ephemeral|standard|regulated", "days": 30}`)
      - `save_state()` MUST write a top-level manifest and the files listed
         therein.
      - `load_state()` MUST validate checksums and `schema_version` before
         rehydration.
3. **No opaque blobs by default.**
      - Opaque binary exports (e.g., pickle) are allowed only behind an explicit
         but keep it non-default.

  available for advanced tooling.

## 3) ADR-031: Calibrator primitives + explainer state persistence

**Goal:** Implement versioned `to_primitive`/`from_primitive` contracts for
calibrators and add explainer `save_state()`/`load_state()`.

**Status (2026-02-23):** Completed. Versioned primitives and save/load round-trip implemented and tested.

### 3.0 Relevant references

- **[RELEASE_PLAN_v1.md](RELEASE_PLAN_v1.md)** — v0.11.0 task definition and release gates.
- **[adrs/ADR-031-calibrator-serialization-and-state-persistence.md](adrs/ADR-031-calibrator-serialization-and-state-persistence.md)** — serialization/persistence contract and versioning expectations.
- [adrs/ADR-009-input-preprocessing-and-mapping-policy.md](adrs/ADR-009-input-preprocessing-and-mapping-policy.md) — mapping snapshots that should round-trip through persistence.
- [adrs/ADR-005-explanation-json-schema-versioning.md](adrs/ADR-005-explanation-json-schema-versioning.md) — versioning principles and JSON-safe constraints.
- [adrs/ADR-002-validation-and-exception-design.md](adrs/ADR-002-validation-and-exception-design.md) — failure modes and exception types.
- [../standards/STD-003-test-coverage-standard.md](../standards/STD-003-test-coverage-standard.md) — persistence round-trip testing expectations.

### 3.1 Current anchors in code

- Calibration modules live under `src/calibrated_explanations/core/` and
   `src/calibrated_explanations/calibration/`.
- Export/import is already a public-facing theme in docs (see foundations).

### 3.2 Implementation steps

1. **Define primitive schema and version policy.**
    - Use JSON-safe primitives only.
    - Require `schema_version` and a calibrator type discriminator.
      - Every calibrator primitive includes: `schema_version`, `calibrator_type`,
         `parameters`, and `checksums`.
      - Validate `schema_version` and fail with actionable errors when unsupported.

2. **Implement calibrator primitives.**
    - For each built-in calibrator, add:
       - `to_primitive() -> dict`
       - `from_primitive(payload: Mapping[str, object]) -> <Calibrator>`
    - Validate schema version range and fail fast with actionable errors.

3. **Implement explainer persistence API.**
    - Add `save_state(path_or_fileobj)` and `load_state(path_or_fileobj)` on the
       wrapper entry point (recommended) so preprocessing (ADR-009) is included.
      - Persist as a directory or archive containing:
         - `manifest.json` (top-level manifest with checksums and schema version)
         - calibrator primitive JSON
         - preprocessing mapping snapshot(s)
         - `explainer_config.json` (plugin identifiers and versions)
         - optional `audit.log`
      - Atomicity/durability requirements:
         - Write to a temp location and finalize using atomic rename/move.
         - Never partially overwrite an existing artifact; on failure, leave the
            previous artifact intact.
         - Include checksums for every file and validate them on load.
      - Implement `load_state()` to validate checksums and `schema_version` before
         rehydrating; on mismatch raise `IncompatibleStateError`.
      - Remote storage upload robustness (where implemented):
         - Prefer a two-phase pattern (upload to staging + checksum verify +
            promote/commit).
         - Enforce RBAC/credential checks before upload; on denial fail fast with
            actionable guidance.

4. **Round-trip tests.**
    - Ensure saved state rehydrates to identical predictions on the same input.
    - Ensure incompatible schema versions raise a documented exception.
      - Add a negative test that mutates a checksum and asserts `load_state()`
         rejects the artifact.

### 3.3 Verification checklist

- Unit tests cover save/load for both classification and regression flows.
- State artifacts are JSON-safe and stable.
- Run `pytest -q`.
