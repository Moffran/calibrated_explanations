## Global rules (deterministic reproducibility + security)

These rules apply across all v0.11.0 work items (domain model, preprocessing,
serialization, persistence, plugin telemetry).

1. **Canonical hashing of manifests.**
         canonicalized deterministically and hashed. Canonicalization rules
         are strict and MUST be followed verbatim when producing the bytes to
         hash:

         - Numbers must be rendered in shortest decimal form (no gratuitous
           exponential notation) where feasible.
         - Datetimes must be serialized as ISO-8601 UTC (`YYYY-MM-DDThh:mm:ssZ`).
         - `null` must use JSON `null`.
         - Canonical JSON MUST be produced by:

           ```py
           canonical = json.dumps(
                 manifest,
                 sort_keys=True,
                 separators=(",",":"),
                 ensure_ascii=False,
           ).encode("utf-8")
           parameters_hash = "sha256:" + sha256(canonical).hexdigest()
           ```

      - Manifests MUST NOT include raw training data or PII. If sensitive keys
         - A configurable denylist and allowlist of payload keys (strings or
         - Provide an explicit redaction helper that replaces sensitive values
           with the literal string `'<redacted>'` before persistence when
           redaction mode is requested. Record `redaction_applied: true` in the
           manifest when redaction occurred.
         - Provide an allowlist override API for known-safe fields; overrides
           must be auditable and recorded in the manifest metadata.
      - Persist the canonical manifest as an immutable artifact and, where
         permitted, reference it from `metadata.audit.config_artifact` (or
         `metadata.audit.run_id` when artifact storage is unavailable). The
         manifest MUST include `access_control` and `retention_policy` fields
         (see artifact manifest schema below) and documentation must explain
         how artifact storage enforces RBAC and how to revoke access.

      - Minimize runtime and build dependencies: prefer Python stdlib code for
         canonicalization, hashing, and validation. Introduce third-party
         libraries only when the functionality cannot be implemented safely and
         deterministically with the stdlib.

2. **Artifact manifest schema.**
      - All persisted artifacts (parameter manifest, calibrator primitives,
         mappings, saved state) MUST include:
         - `schema_version`
         - `access_control` (e.g., `{"read": ["role:maintainer"], "write": ["role:ci"]}`)
         - `retention_policy` (e.g., `{"class": "ephemeral|standard|regulated", "days": 30}`)
      - `save_state()` MUST write a top-level manifest and the files listed
         therein.
      - `load_state()` MUST validate checksums and `schema_version` before
         rehydration.
3. **No opaque blobs by default.**
      - Opaque binary exports (e.g., pickle) are allowed only behind an explicit
         but keep it non-default.

## 5) Conformal Guard

**Goal:** Ensure guarded explanations filter out unrealistic perturbations using
conformal in-distribution testing over perturbed candidate intervals.

### 5.0 Implementation

The release task is implemented via the guarded explanation runtime, not a
separate `conformal_guard` parameter.

1. Public API surface:
   - `CalibratedExplainer.explain_guarded_factual(...)`
   - `CalibratedExplainer.explore_guarded_alternatives(...)`
2. Both methods delegate to the shared guarded core
   (`core/explain/_guarded_explain.py::guarded_explain`).
3. `guarded_explain` uses `InDistributionGuard` (KNN-based conformal
   non-conformity / p-value testing) to decide whether each perturbed bin
   representative is conforming.
4. Non-conforming bins are excluded from emitted rules. Optional
   `merge_adjacent=True` merges conforming neighboring bins, then re-validates
   merged representatives with the same guard.
5. Outputs are guarded explanation classes
   (`GuardedFactualExplanation`, `GuardedAlternativeExplanation`) and expose
   interval-level audit payloads via `get_guarded_audit()`.

### 5.1 Verification checklist

- Guarded entry points execute end-to-end and return guarded explanation
  containers/types.
- Guard decisions are based on conformal p-values (`p_value >= significance`),
  with optional Bonferroni adjustment (`use_bonferroni=True`).
- Non-conforming perturbation intervals are filtered from emitted rules.
- `get_guarded_audit()` includes deterministic interval-level conforming/non-
  conforming traces and summary counts.

### 6.1 What is currently implemented

The following components are complete and covered by existing tests

#### 6.1.1 Explanation generation via `multi_labels_enabled=True`

`CalibratedExplainer.explain_factual(x, multi_labels_enabled=True)` and
`CalibratedExplainer.explore_alternatives(x, multi_labels_enabled=True)` produce a
`MultiClassCalibratedExplanations` container.

The orchestrator (`core/explain/orchestrator.py`, `invoke_factual` / `invoke_alternative`)
detects `multi_labels_enabled=True` and iterates over `np.unique(y_cal)` (extended to
`class_labels` length when provided), calling the legacy explain path once per class with
`labels=[cls, ...]` pinned. Each per-class result is stored as
`multi_label_explanations[i][cls] = explanation`, yielding a
`list[dict[class_key, Explanation]]` structure.

When calibrators define more classes than are present in `y_cal` (e.g., a class with zero
calibration instances), the range is expanded to `np.arange(len(class_labels))`.

#### 6.1.2 `MultiClassCalibratedExplanations` container

Defined in `explanations/explanations.py`. Key contracts:

- `__init__`: accepts `explanations: list[dict[int, Explanation]] | None` and deep-copies on
  construction to avoid shared-state mutations.
- `__getitem__(int)` → single-instance `MultiClassCalibratedExplanations` (mirrors
  `CalibratedExplanations` semantics).
- `__getitem__(slice | list | ndarray)` → multi-instance sub-collection (boolean and integer
  index arrays both supported).
- `__getitem__((int, int))` → the per-class `FactualExplanation` or `AlternativeExplanation`.
- `__getitem__((int, str))` → lookup by class-label string (resolves via `get_class_labels()`).
- `get_explanation(index, class_idx=None)` → public accessor delegating to `__getitem__`.
- `__repr__` → class-by-class human-readable summary.
- `_first_explanation_for_instance(index)` → internal helper for dispatch in visualization.
- `X_test` property → backward-compatible alias for `x_test`.
- `num_classes` attribute.

#### 6.1.3 Multiclass visualization

`MultiClassCalibratedExplanations.plot()` dispatches to `plot_factual()` or
`plot_alternative()` depending on which explanation type is stored. All three methods carry
`# pragma: no cover  # ADR-023` annotation, since they require a running matplotlib backend
and are exercised in `@pytest.mark.viz` integration tests instead.

`plot_factual()` collects all per-class rules, calls `sort_factuals_by_rule()` to align rules
by rule text across classes, and delegates to `_plot_probabilistic_dict()` for the stacked
multi-class bar chart.

`plot_alternative()` calls `_plot_alternative_dict()` for the stacked multi-class alternative
view. Unlike binary mode, the `style='triangular'` path deliberately raises a
`ValidationError` because triangular plots are not meaningful for all-classes views. This
is documented behavior.

`merge_rules()` and `sort_factuals_by_rule()` are utility helpers; `merge_rules` currently
has zero callers and is marked `# pragma: no cover  # dead code`.

#### 6.1.4 Default (non-multi-label) multiclass behaviour

When `multi_labels_enabled=False` (the default), multiclass models produce standard
`CalibratedExplanations` / `AlternativeExplanations` collections where each explanation
corresponds to the **argmax class** (i.e. the predicted class). This is documented in
`docs/tasks/classification.md` and `docs/get-started/quick_api.md`.

### 6.3 Proposed hardening (open items)

The items below are gaps where `MultiClassCalibratedExplanations` diverges from the
behaviour users get with binary or regression explanations. Each should be addressed to
make multiclass a **first-class citizen** of the CE API.

#### 6.3.1 Collection-mutating methods inherited but broken for dict explanations

**Problem:** `CalibratedExplanations.explanations` is assumed by many inherited methods
to be a `list[Explanation]`. In `MultiClassCalibratedExplanations`, it is a
`list[dict[int, Explanation]]`. The following inherited methods silently fail or raise an
`AttributeError` at runtime when called on a `MultiClassCalibratedExplanations` instance:

- `add_conjunctions()` / `remove_conjunctions()` — call `explanation.add_conjunctions()` on dicts.
- `filter_rule_sizes()` / `filter_features()` — call these on dict items.
- `get_rules()` — calls `explanation.get_rules()` on dicts.
- `reset()` — calls `explanation.reset()` on dicts.
- `as_lime()` / `as_shap()` — iterate over `self.explanations` expecting explanation objects.
- `predictions` / `probabilities` / `prediction_interval` / `predict` properties — access
  `.predict`, `.prediction_probabilities`, etc. on dicts.
- `to_narrative()` / `to_dataframe()` — the narrative plugin iterates over flat explanations.

**Fix:** Override each of these in `MultiClassCalibratedExplanations` to dispatch through
the dict values for every instance:

```python
def add_conjunctions(self, n_top_features=5, max_rule_size=2, **kwargs):
    for class_dict in self.explanations:
        for explanation in class_dict.values():
            explanation.add_conjunctions(n_top_features, max_rule_size, **kwargs)
    return self

def get_rules(self):
    return [
        {cls_key: exp.get_rules() for cls_key, exp in class_dict.items()}
        for class_dict in self.explanations
    ]
```

The same pattern applies to `remove_conjunctions`, `reset`, `filter_rule_sizes`,
`filter_features`. For `as_lime`, `as_shap`, and `to_narrative`, raise an explicit
`NotImplementedError` with a descriptive message until a multiclass-aware adapter exists,
rather than silently returning wrong results.

**Test rubric:** Add unit tests in
`tests/unit/explanations/test_multiclass_calibrated_explanations.py`:
- `test_should_propagate_add_conjunctions_to_all_classes_when_called_on_collection`
- `test_should_propagate_filter_features_to_all_classes_when_called_on_collection`
- `test_should_return_per_class_rules_dict_when_get_rules_called`
- `test_should_raise_not_implemented_when_as_lime_called`

#### 6.3.2 Serialization: `to_json` / `from_json` not multiclass-aware

**Problem:** `MultiClassCalibratedExplanations` inherits `to_json()`, `to_json_stream()`,
and `from_json()` from `CalibratedExplanations`. All three assume `self.explanations` holds
flat `Explanation` objects and pass each to `_legacy_payload(exp)`, which will fail
(or silently serialize only `{}`) when `exp` is a dict.

**Fix:** Override `to_json()` on `MultiClassCalibratedExplanations` to serialize the
per-class structure:

```python
# Recommended schema extension for multiclass:
{
    "schema_version": "1.0.0",
    "collection": { ... },  # same as base
    "multiclass": true,
    "num_classes": 3,
    "explanations": [
        {
            "instance_index": 0,
            "per_class": {
                "0": { /* schema-v1 explanation payload for class 0 */ },
                "1": { /* schema-v1 explanation payload for class 1 */ },
                "2": { /* schema-v1 explanation payload for class 2 */ }
            }
        }
    ]
}
```

Ensure `from_json()` can round-trip this format back to a
`MultiClassCalibratedExplanations` instance. Add `schema_version` validation. Align with
the artifact manifest schema (§Global rules §2: `schema_version`, `access_control`,
`retention_policy`).

**Test rubric:**
- `test_should_round_trip_multiclass_to_json_and_from_json_when_factual_explanations_present`
- `test_should_round_trip_multiclass_to_json_and_from_json_when_alternative_explanations_present`
- `test_should_raise_validation_error_when_from_json_receives_wrong_schema_version`

#### 6.3.3 `features_to_ignore` not propagated in multi-label branch

**Problem:** In `orchestrator.invoke_factual` and `invoke_alternative`, the
`multi_labels_enabled` branch calls:

```python
explanations = legacy_explain(
    self.explainer, x, threshold, low_high_percentiles, bins, labels=labels
)
```

The `features_to_ignore` argument (set earlier via `set_discretizer`) is not forwarded to
`legacy_explain`. The binary / non-multi-label path does forward it:

```python
return legacy_explain(
    ...,
    features_to_ignore=features_to_ignore,
    ...
)
```

This means that features listed in `features_to_ignore` are excluded from the discretizer
but not from the explain loop itself in the multiclass path.

**Fix:** Pass `features_to_ignore=features_to_ignore` and
`interval_summary=kwargs.get("interval_summary", self.explainer.interval_summary)` as
keyword arguments in both multiclass `legacy_explain` calls in `orchestrator.py`.

**Test rubric:**
- `test_should_respect_features_to_ignore_in_multiclass_multi_labels_enabled_path`
  (unit test in `tests/unit/core/explain/test_explain_orchestrator.py`).

#### 6.3.4 `reject_policy` not applied in multi-label branch

**Problem:** The `multi_labels_enabled` branch short-circuits before the `self.invoke(...)`
call that would propagate `reject_policy`. Users calling
`explain_factual(x, multi_labels_enabled=True, reject_policy=...)` silently get an
explanation without any reject-policy guard applied.

**Fix:** Before the per-class loop in the multi-label branch, evaluate the reject policy
(if any) and apply it to the full batch of `x`. Alternatively, document this limitation
prominently and raise a `NotImplementedError` with a clear message when
`reject_policy is not None and multi_labels_enabled is True`.

**Test rubric:**
- `test_should_warn_or_raise_when_reject_policy_combined_with_multi_labels_enabled`

#### 6.3.5 `multi_labels_enabled=True` on binary data — guard or correct behavior

**Problem:** No validation prevents `multi_labels_enabled=True` from being used on a
binary dataset. Behaviour is technically correct (2-class dict per instance) but may
surprise users who expect the binary mode to remain unchanged.

**Fix:** Add a guard in `invoke_factual` / `invoke_alternative`: if `multi_labels_enabled`
is `True` but `len(classes) < 3`, emit a `UserWarning` (consistent with the Fallback
Visibility Policy in §7 of `copilot-instructions.md`) explaining that `multi_labels_enabled`
is only useful for 3+ class problems and the binary path will be used instead, OR simply
allow it but add a `LOG.info` trace. The warning approach is mandatory per the fallback
policy.

**Test rubric:**
- `test_should_warn_when_multi_labels_enabled_on_binary_dataset`

#### 6.3.6 Numeric class-key type consistency

**Problem:** Dict keys are produced by `np.unique(self.explainer.y_cal)` and are
`np.int64` (or the dtype of `y_cal`). The `__getitem__` branch checks
`isinstance(class_idx, int)`, which is `False` for `np.int64` in numpy ≥ 2.0. While dict
__lookup__ works (because `hash(np.int64(0)) == hash(0)` and equality holds), the
dispatch branch:

```python
if isinstance(class_idx, int):
    return self.explanations[index].get(class_idx, None)
```

fails silently (falls through to `raise ValidationError`) when `class_idx` is `np.int64`.

**Fix:** Normalize the isinstance check to also accept `np.integer`:

```python
if isinstance(class_idx, (int, np.integer)):
    return self.explanations[index].get(int(class_idx), None)
```

Also normalize the stored keys to plain Python `int` at construction time (in `invoke_factual`
and `invoke_alternative`), so the container is always indexed by `int`.

**Test rubric:**
- `test_should_return_explanation_when_class_idx_is_numpy_integer`
- `test_should_have_int_keys_in_explanations_dict_when_constructed`

#### 6.3.7 `__len__` and iteration protocol for dict-structured explanations

**Problem:** `len(multi_explanations)` returns the number of *instances* (correct).
However, `for exp in multi_explanations` iterates over `list[dict]`, yielding dicts.
Code that calls `for exp in collection` and then accesses `exp.feature_weights` (standard
pattern in binary) would break silently for multiclass containers.

**Fix:** Override `__iter__` to yield single-instance `MultiClassCalibratedExplanations`
views (mirroring the `int`-indexing contract):

```python
def __iter__(self):
    for i in range(len(self.explanations)):
        yield self[i]
```

This aligns multiclass iteration with binary/regression `CalibratedExplanations.__iter__`
(which yields per-instance sub-collections).

**Test rubric:**
- `test_should_yield_single_instance_multiclass_collection_when_iterated`

#### 6.3.8 `to_narrative` / `to_dataframe` compatibility

**Problem:** The narrative plugin (`viz/narrative_plugin.py`) uses a flat
`for exp in collection.explanations` loop. For `MultiClassCalibratedExplanations`
this iterates over dicts and produces wrong output without error.

**Fix (near-term):** Override `to_narrative` and `to_dataframe` to raise a clear
`NotImplementedError("to_narrative is not yet supported for multi_labels_enabled=True "
"collections; call get_explanation(i, cls).to_narrative() per class instead.")`.

**Fix (long-term):** Add a multiclass-aware narrative mode that emits one table per
class, or a merged table with a `class` column.

**Test rubric:**
- `test_should_raise_not_implemented_when_to_narrative_called_on_multiclass_collection`

#### 6.3.9 Visualization coverage: lift `# pragma: no cover` where possible

All three plot methods carry `# pragma: no cover  # ADR-023`. ADR-023 allows
matplotlib-heavy paths to be excluded from line coverage provided they are exercised by
`@pytest.mark.viz` integration tests. However, the *logic* portions of `plot_factual` and
`plot_alternative` — rule sorting, `sort_factuals_by_rule`, feature ranking, list
comprehensions — can and should be unit-tested in isolation by mocking the terminal
`_plot_probabilistic_dict` / `_plot_alternative_dict` calls. Only the matplotlib rendering
calls themselves need the ADR-023 exemption.

**Fix:** Extract the rule-aggregation and feature-ranking logic from `plot_factual` /
`plot_alternative` into testable private methods (e.g.,
`_build_factual_plot_payload(class_explanations)`) and unit-test those helpers. The actual
`plt.*` calls remain under `# pragma: no cover`.

Additionally, `merge_rules` is currently dead code (zero callers). Remove or mark with
`# TODO: remove after v0.11.0` to avoid carrying untested surface area.

**Test rubric:**
- `test_should_sort_factuals_by_rule_correctly_when_two_classes_share_same_rule`
- `test_should_build_factual_plot_payload_without_matplotlib_when_class_explanations_provided`
