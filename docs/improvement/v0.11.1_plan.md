# v0.11.1 Release Task Implementation Plan

This plan expands the v0.11.1 tasks from `docs/improvement/RELEASE_PLAN_v1.md`
with implementation details. v0.11.1 is a **registry hardening** release:
no public API changes, no new features — purely internal correctness and
observability improvements that could not ship in v0.11.0 because they require
the `PluginManager` shell introduced there.

## Source references reviewed

- Release task list: `RELEASE_PLAN_v1.md` v0.11.1 section.
- ADRs: ADR-006 (plugin trust model), ADR-011 (deprecation policy),
  ADR-012 (docs/gallery build policy), ADR-027/028 (observability).
- v0.11.0 plan: `v0.11.0_plan.md` §§9–10 (registry improvements + public-surface cleanup prerequisites for tasks 1–5).
- Deferred from v0.11.0 (no API contract changes): ADR-012, ADR-027/028,
   Standard-001 completion, and governance-dashboard hardening are covered here
   as tasks 6–9.
- Registry module: `src/calibrated_explanations/plugins/registry.py`.

## 1) Migrate resolution logic into `PluginManager` (ADR-006)

**Goal:** Make `PluginManager` the single, authoritative source of truth for
plugin resolution, ending the current state where ADR-006's mandate is satisfied
only in name (the `PluginManager` shell from v0.11.0 still delegates non-trivially
to `CalibratedExplainer` internals).

### 1.1 Relevant references

- ADR-006 §Decision: *"The `PluginManager` serves as the single source of truth
  for plugin resolution and defaults."*
- ADR-026 §1: plugin resolution precedence and `_invoke_explanation_plugin` contract.
- ADR-013 §3: interval plugin configuration and fallback chains.
- ADR-015 §5: explanation plugin registry metadata and dependency coordination.

### 1.2 Current anchors in code

- Explanation resolution: `CalibratedExplainer._invoke_explanation_plugin`,
  `_ensure_explanation_plugin`, `_resolve_explanation_plugin_for_mode`.
- Interval resolution: `CalibratedExplainer._resolve_interval_plugin`.
- Plot resolution: `CalibratedExplainer._resolve_plot_plugin`.
- Manager shell (v0.11.0): `src/calibrated_explanations/plugins/manager.py`.

### 1.3 Implementation steps

1. **Move `_resolve_explanation_plugin_for_mode`** from `CalibratedExplainer`
   into `PluginManager.resolve_explanation_plugin(mode, *, task, overrides)`.
   Resolution precedence (keyword → env → pyproject → fallback → default) must
   be preserved exactly per ADR-026.

2. **Move `_resolve_interval_plugin`** into
   `PluginManager.resolve_interval_plugin(*, mode, fast, overrides)`.
   Fallback chain semantics per ADR-013 must be preserved.

3. **Move `_resolve_plot_plugin`** into
   `PluginManager.resolve_plot_plugin(*, style, overrides)`.

4. **Update `CalibratedExplainer`** to delegate all three resolution calls to
   `_plugin_manager.resolve_*()`. `CalibratedExplainer` must not contain
   resolution logic after this change.

5. **Update `_invoke_explanation_plugin`** to call
   `_plugin_manager.resolve_explanation_plugin(...)` and then invoke the
   result, keeping the bridge-monitor and invariant-check logic in
   `CalibratedExplainer` (execution orchestration stays there; resolution moves).

6. Wire `PluginManager` construction into `CalibratedExplainer.__init__`:
   ```python
   self._plugin_manager = PluginManager(policy=self._trust_policy)
   ```

### 1.4 Verification checklist

- All existing factual/alternative/FAST integration tests pass unchanged.
- `CalibratedExplainer` contains no `_resolve_*_plugin` methods after the migration.
- Custom `PluginTrustPolicy` injected at `CalibratedExplainer` construction is
  used during resolution via `_plugin_manager`.
- `pytest -q` green; no coverage regression.

## 2) Atomic trust state (ADR-006)

**Goal:** Eliminate the divergence risk between descriptor `trusted` field and
`_TRUSTED_*` identifier sets.

### 2.1 Problem

Two sources of truth for trust exist in `registry.py`:
- The `trusted: bool` field on each `*PluginDescriptor` dataclass.
- The `_TRUSTED_EXPLANATIONS`, `_TRUSTED_INTERVALS`, `_TRUSTED_PLOT_BUILDERS`,
  `_TRUSTED_PLOT_RENDERERS` sets of identifier strings.

`_update_trust_keys` mutates raw metadata dicts but does not always follow up
with a set update, and raw `register()` calls that skip the descriptor path
can leave the sets stale. This means `find_*_trusted(id)` (reads descriptor)
and `list_*_descriptors(trusted_only=True)` (reads set) can disagree.

### 2.2 Implementation steps

1. **Centralise all trust state mutations in `PluginManager`** via a single
   internal `_set_trust(identifier, trusted, *, store, trusted_set, plugin_store)`.
   This method updates descriptor, set, and raw metadata atomically.

2. **Replace all call sites** in `registry.py` that mutate trust state
   (`_update_trust_keys`, `_TRUSTED_*.add/discard`, descriptor reconstruction)
   with calls to `PluginManager._set_trust(...)`.

3. **Add an invariant assertion** (debug-only, gated on `__debug__`) that runs
   after any trust mutation and verifies that the descriptor's `trusted` field
   matches set membership for all registered identifiers.

4. **Update `mark_explanation_trusted` / `mark_interval_trusted` etc.** to
   route through `PluginManager._set_trust`.

### 2.3 Verification checklist

- After `mark_explanation_trusted(id)`: `find_explanation_descriptor(id).trusted
  is True` and `id in list_explanation_descriptors(trusted_only=True)`.
- After `mark_explanation_untrusted(id)`: both are consistently `False`/absent.
- Repeat for interval, plot builder, and plot renderer variants.
- No existing test exercises inconsistent state (add a regression test asserting
  the invariant explicitly for each plugin category).

## 3) Complete ADR-028 governance audit coverage

**Goal:** Add structured audit events for every accepted plugin registration;
currently only deny/skip paths use `calibrated_explanations.governance.*`.

### 3.1 Relevant references

- ADR-028 §2: governance vs operational separation; all trust decisions must
  emit governance events.
- copilot-instructions.md §7 Fallback Visibility Policy: every trust/registry
  decision must be observable.

### 3.2 Current state

`_warn_untrusted_plugin` and the deny branch of `load_entrypoint_plugins` emit
`calibrated_explanations.governance.plugins` events, but:
- `register_explanation_plugin` and `register_interval_plugin` log nothing for
  accepted trusted registrations.
- `register_plot_builder` and `register_plot_renderer` log nothing at all.
- The `register()` legacy path logs nothing.

### 3.3 Implementation steps

1. **Add a `_emit_governance_event` helper** that writes a structured governance
   log entry via `logging.getLogger("calibrated_explanations.governance.registry")`
   with fields `identifier`, `source`, `trusted`, `decision`
   (`"accepted"` | `"denied"` | `"skipped_untrusted"`).

2. **Call `_emit_governance_event`** at the end of every `register_*` function
   after trust determination, regardless of accept/deny outcome.

3. **Standardise the governance logger name** to
   `calibrated_explanations.governance.registry` throughout `registry.py` and
   `load_entrypoint_plugins`; retire the inconsistent use of
   `calibrated_explanations.governance.plugins` inside `registry.py` (the
   plugins sub-logger may still be used by other modules such as `builtins.py`).

4. **Add a test** `tests/observability/test_governance_registry_accepted.py`
   asserting that accepted registrations emit a governance log at `INFO` level
   with the required fields.

### 3.4 Verification checklist

- `register_explanation_plugin(id, plugin, source="builtin")` emits an
  `INFO` governance event with `decision="accepted"` and `trusted=True`.
- `register_explanation_plugin(id, plugin, source="entrypoint")` for an
  untrusted, denied, and accepted plugin each emit distinct governance events.
- The governance logger name is `calibrated_explanations.governance.registry`
  consistently inside `registry.py`.

## 5) Legacy `_REGISTRY`/`_TRUSTED` list deprecation (ADR-011)

**Goal:** Start the ADR-011 two-minor-release removal clock on the legacy
list-based registry, giving a v0.13.0 sunset target.

### 5.1 Current state

`_REGISTRY` (list of `ExplainerPlugin`) and `_TRUSTED` (list of trusted plugins)
are maintained in parallel with the identifier-keyed descriptor dicts. The lists
are only needed by `register()`, `trust_plugin()`, `untrust_plugin()`,
`find_for()`, `find_for_trusted()`, and `list_plugins()`.

### 5.2 Implementation steps

1. **Add `deprecate()` calls** to `register()`, `trust_plugin()`,
   `untrust_plugin()`, `find_for()`, `find_for_trusted()`, and `list_plugins()`
   with message:
   ```
   "The list-based registry path (register/trust_plugin/find_for) is deprecated
   and will be removed in v0.13.0. Use register_*_plugin / find_*_plugin /
   list_*_descriptors instead."
   ```
   Use the central `deprecate()` helper from `utils.deprecations` with unique
   `key` per function.

2. **Document the migration** in `docs/migration/deprecations.md`:
   - Table row: `register()` → `register_explanation_plugin()` / `register_interval_plugin()` / `register_plot_builder()`.
   - Table row: `trust_plugin()` → `mark_explanation_trusted()` / `mark_interval_trusted()`.
   - Table row: `find_for()` → `list_explanation_descriptors()` filtered by capability.
   - Note that v0.13.0 removes these symbols entirely.

3. **Update in-tree callers** — `register_explanation_plugin` currently calls
   `register(plugin, ...)` for backwards compat. Replace with direct descriptor
   dict + set updates so no in-tree code emits the deprecation warning.

4. **Add a CI deprecation test** that verifies calling `register(plugin)` emits
   the expected deprecation warning with `pytest.warns(DeprecationWarning)` and
   that setting `CE_DEPRECATIONS=error` causes it to raise.

### 5.3 Verification checklist

- `register(plugin)` emits a `DeprecationWarning` containing "v0.13.0".
- `find_for(model)` emits a `DeprecationWarning`.
- No in-tree production code triggers the deprecation warning (tests against
  `CE_DEPRECATIONS=error` in CI).
- Migration table in `docs/migration/deprecations.md` updated.

## 6) ADR-012: Notebook/gallery execution + time ceilings

**Goal:** Document the tooling choice and enforce notebook execution/time
ceilings in docs CI. (Deferred from v0.11.0 — no API contract change.)

### 6.0 Relevant references

- [adrs/ADR-012-documentation-and-gallery-build-policy.md](adrs/ADR-012-documentation-and-gallery-build-policy.md) — notebook/gallery execution policy and time ceilings.
- [adrs/ADR-010-core-vs-evaluation-split-and-distribution.md](adrs/ADR-010-core-vs-evaluation-split-and-distribution.md) — keeps heavy/optional deps out of core installs.
- [../standards/STD-004-documentation-audience-standard.md](../standards/STD-004-documentation-audience-standard.md) — doc structure and audience expectations.

### 6.1 Implementation steps

1. **Execution engine (`nbconvert` MVP).** Use `nbconvert` invoked from docs CI
   to execute notebooks. Wrap execution in a stable driver script that sets
   per-notebook/per-cell timeouts via env vars, captures output, and emits a
   JSON execution report. Pin the docs execution environment for release branches.
   Document a migration path to `sphinx-gallery` in a later release.

2. **Enforce time ceilings.** Configure per-notebook and per-cell timeouts;
   failures are blocking on release branches per ADR-012. Define a standard
   metadata tag (e.g. `tags: ["ci-skip", "slow"]`) for long-running examples.

3. **Make optional dependencies explicit.** Docs execution jobs install the
   right extras (viz/notebooks); core unit tests must not require them (ADR-010).

### 6.2 Decision

- Use `nbconvert` as the MVP execution engine with explicit timeouts.
- Document a later migration path to `sphinx-gallery`.

### 6.3 Verification checklist

- Docs CI runs notebook execution with explicit timeouts.
- A deliberately failing example causes the docs job to fail.
- A smoke test covers the notebook execution driver behavior.
