## 6) Deep memory audit and retention hardening

### Goal
Deliver bounded-memory behavior for long-lived processes with explicit lifecycle semantics and measurable RSS stabilization.

### Why this task exists
The audit identified retention growth through explanation retention, optional integration caches, and executor lifecycle state.

### Governing references
- ADR-003 (caching strategy)
- ADR-004 (parallel lifecycle)
- ADR-028 (observability for retention events)

### Hardening findings incorporated (binding design choices)
1. **`latest_explanation` design:** retain only the most recent explanation object and clear it in `reset()` and `close()`.
2. **SHAP/LIME caching design:** keep one cached helper instance each (`LimeHelper`, `ShapHelper`) and require explicit reset on `CalibratedExplainer.reset()`.
3. **Plugin/cache bounding design:** bound plugin manager explanation-instance cache to max 16 entries with LRU eviction.
4. **Parallel path resolution:** authoritative path is `src/calibrated_explanations/parallel/parallel.py` (not `core/parallel/parallel.py`).
5. Numeric RSS thresholds below are **v0.11.2 plan policy gates** derived from the deep-memory audit objective; they are not ADR-stated constants.

### Compatibility posture
Additive lifecycle hardening (`reset` semantics clarified) with no API removals.

- No cross-process shared-memory retention coordination.
