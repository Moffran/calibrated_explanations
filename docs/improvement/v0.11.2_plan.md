# v0.11.2 Release Task Implementation Plan

This document is the execution plan for all `v0.11.2 (config hardening and ADR governance sweep)` tasks listed in `docs/improvement/RELEASE_PLAN_v1.md`.


## Milestone control posture

- This plan is detailed for continuity but is not the active control surface while another milestone is active.
- Preserve prescriptive execution detail as-is; re-baseline only at milestone boundaries when this milestone becomes active.

## Planning standard used in this file

Each task contains:

1. Why the task exists (ADR/standard alignment).
2. Concrete code and doc touchpoints.
3. Ordered implementation steps with dependency order.
4. Required tests/checks and release artifacts.
5. A detailed verification checklist that can be executed as-is.

## Global execution constraints

- Keep CE-first and ADR-first behavior.
- Preserve public API compatibility — all changes in this milestone are internal migration or documentation only.
- Any behavior change must include docs + tests in same PR.
- Validation path for each task PR:
  1. `make local-checks-pr`
  2. Task-targeted pytest runs
  3. `make local-checks` if task changes branch-gate behavior

## 1) ConfigManager Phase B migration (ADR-034)

Goal: migrate the four remaining allowlisted runtime modules to `ConfigManager`, removing them from the CI enforcement allowlist so the allowlist reaches zero.

Why: ADR-034 is now Accepted and binding. Phase B assumes the ADR-034 architectural contract and the Phase A delivery defined in `v0.11.1_plan.md`, and completes migration of the remaining allowlisted runtime modules that still read `os.getenv` or parse `pyproject.toml` directly.

ADR-034 boundary note:
- Phase B is migration closure for remaining allowlisted runtime modules and allowlist-zero enforcement.
- It must not restate or re-open ADR-034 status promotion work; that status is already `Accepted`.

Primary touchpoints:

- `src/calibrated_explanations/core/cache/cache.py`
- `src/calibrated_explanations/core/parallel/parallel.py`
- `src/calibrated_explanations/core/explain/_feature_filter.py`
- `src/calibrated_explanations/core/prediction/orchestrator.py`
- `scripts/quality/check_config_manager_usage.py` (the allowlist script)
- `tests/unit/core/test_config_manager.py`

Step-by-step implementation:

1. Audit each allowlisted module.
   1. For each of the four modules, list every `os.getenv`, `os.environ`, and direct `pyproject.toml` read.
   2. Map each read to the corresponding `ConfigManager` key defined by ADR-034 and the Phase A implementation.
2. Migrate `cache/cache.py`.
   1. Replace direct env reads with `ConfigManager` accessor calls.
   2. Update cache initialization path to use resolved config values.
   3. Add/update unit tests in `tests/unit/core/` to exercise the migrated path.
3. Migrate `parallel/parallel.py`.
   1. Replace direct env reads with `ConfigManager` accessor calls.
   2. Ensure `CE_PARALLEL` and related env overrides are honored via manager.
   3. Add/update unit tests.
4. Migrate `core/explain/_feature_filter.py`.
   1. Replace direct env/config reads with `ConfigManager` accessor calls.
   2. Add/update unit tests.
5. Migrate `core/prediction/orchestrator.py`.
   1. Replace direct env/config reads with `ConfigManager` accessor calls.
   2. Add/update unit tests.
6. Shrink the allowlist.
   1. Remove each migrated module from `check_config_manager_usage.py` allowlist.
   2. Run the check script to confirm zero violations.

Verification checklist:

- [ ] `cache.py` reads no `os.getenv`/`os.environ` directly; all config via `ConfigManager`.
- [ ] `parallel.py` reads no `os.getenv`/`os.environ` directly; all config via `ConfigManager`.
- [ ] `_feature_filter.py` reads no `os.getenv`/`os.environ` directly; all config via `ConfigManager`.
- [ ] `orchestrator.py` reads no `os.getenv`/`os.environ` directly; all config via `ConfigManager`.
- [ ] `scripts/quality/check_config_manager_usage.py` allowlist is empty (or contains only explicitly sanctioned boundaries with documented justification).
- [ ] `python scripts/quality/check_config_manager_usage.py` reports zero violations.
- [ ] Unit tests for each migrated module pass and cover the new config-resolution path.
- [ ] `make local-checks-pr` passes.

## 4) Governance dashboard extension (lint + preprocessing/domain telemetry)

Goal: surface lint status and preprocessing/domain telemetry through one documented, machine-readable workflow so maintainers can inspect the project's governance posture from a single artifact.

Why: ADR-028 requires structured governance observability; v0.11.1 Tasks 3 and 7 establish the plugin `GovernanceEvent` schema and the domain-logger enforcement tooling; v0.11.1 Task 20 adds the parallel `governance_config_event_schema_v1.json` for ConfigManager lifecycle events. This task builds on those foundations by generating a machine-readable governance status artifact that aggregates lint signals and preprocessing/domain telemetry. Delivering this in v0.11.2 (after the v0.11.1 schemas and tooling are stable) avoids coupling dashboard implementation to in-flight v0.11.1 schema work. Relocating from v0.11.1 follows the governance sweep theme of v0.11.2.

Contract boundary:
- This task is an aggregation/reporting surface only.
- It may summarize CI outcomes and counts derived from the runtime governance events defined in v0.11.1 Tasks 3 and 20.
- It must not define a replacement runtime governance-event contract, logger vocabulary, or schema for plugin/config lifecycle events.

Primary touchpoints:

- Governance docs/hub pages
- CI artifact generation scripts/workflows (`.github/workflows/`)
- `scripts/quality/` (lint status summary emission)
- Telemetry documentation and schema examples
- `docs/observability/governance_status_artifact.md` (new)

Step-by-step implementation:

1. Define governance status artifact schema.
   1. Fields: `component`, `check_name`, `status`, `timestamp`, `details_url`, optional `invocation_id`.
   2. Version schema (e.g., `schema_version: "1.0"`) and document compatibility expectations.
   3. Commit schema definition to `docs/observability/governance_status_artifact.md`.
   4. State explicitly that this schema is a derived CI/reporting artifact, not a runtime governance-event schema.
2. Implement artifact generation.
   1. Extend the CI lint check step to emit a `governance_status.json` artifact summarizing lint check outcomes per component.
   2. Emit a preprocessing/domain telemetry summary block for selected release checks (e.g., input-mapping validation counts, plugin-resolution telemetry), preferably by summarizing existing governance/telemetry outputs rather than inventing new runtime event payloads.
   3. Ensure artifact is uploaded as a CI build artifact so it is retrievable per run.
3. Document retrieval workflow.
   1. Document where the artifact is published (CI artifacts tab, S3 bucket, or equivalent).
   2. Document how maintainers fetch and interpret it (CLI command or browser link).
4. Document privacy/reliability constraints.
   1. Add data minimization guidance: no PII, no user input values in artifact.
   2. Add sampling and retention guidance (e.g., retain last N release-branch runs).
5. Add an example ingestion command.
   1. Provide a `jq` or Python snippet for extracting pass/fail status from the artifact.

Verification checklist:

- [ ] Artifact schema is documented with versioning notes in `docs/observability/governance_status_artifact.md`.
- [ ] CI produces `governance_status.json` artifact on relevant workflow runs.
- [ ] Lint and telemetry signals are represented in the artifact output.
- [ ] Docs state explicitly that `governance_status.json` is a derived CI artifact and does not supersede the runtime governance schemas from v0.11.1 Tasks 3 and 20.
- [ ] Maintainer retrieval/interpretation workflow is documented.
- [ ] Privacy and retention guidance is documented.
- [ ] `make local-checks-pr` passes.

## 6) Deep memory audit and retention hardening

Goal: Address memory growth in long-lived server/online sessions by fixing unbounded retention paths, integration leaks, and stale resource accumulation identified in the 2026-03-13 audit.

Why: The audit identified that while the framework has no traditional "leaks", it retains significant state (last explanations, SHAP/LIME explainers, worker pools) that causes memory to grow linearly with request count or mode switching in long-lived processes. Fixes are required to ensure server-side stability.

Primary touchpoints:

- `src/calibrated_explanations/core/calibrated_explainer.py`
- `src/calibrated_explanations/integrations/shap.py`
- `src/calibrated_explanations/integrations/lime.py`
- `src/calibrated_explanations/parallel/parallel.py`
- `src/calibrated_explanations/viz/matplotlib_adapter.py`
- `src/calibrated_explanations/plugins/manager.py`

Step-by-step implementation:

1. **Implement `reset()` and `close()` APIs.**
   1. Add `CalibratedExplainer.reset(clear_calibration_data=False)` to purge `latest_explanation`, helper caches, and plugin instances.
   2. Ensure `CalibratedExplainer.close()` explicitly shuts down any active `ParallelExecutor` pool.
2. **Harden Integration Helpers.**
   1. Update `ShapHelper.reset()` and `LimeHelper.reset()` to explicitly nullify `_explainer_instance` and `_reference_explanation`.
   2. Implement optional "ephemeral mode" for integrations where explainers are not cached across calls.
3. **Fix `latest_explanation` Retention.**
   1. Change `self.latest_explanation` to use a weak reference or provide a configuration flag to disable this retention.
4. **Matplotlib Resource Management.**
   1. Audit `matplotlib_adapter.py` to ensure `plt.close(fig)` is called in all non-returning render paths.
5. **Worker Pool Lifecycle.**
   1. Ensure `ParallelExecutor` initialized via `initialize_pool` is tracked for mandatory shutdown in `explainer.close()`.
6. **Plugin Cache Bounding.**
   1. Introduce a maximum size or TTL for `PluginManager` instance/context caches if used in high-churn environments.

Verification checklist:

- [ ] `CalibratedExplainer.reset()` successfully clears `latest_explanation` and integration caches.
- [ ] `CalibratedExplainer.close()` shuts down `ProcessPoolExecutor` workers.
- [ ] `SHAP`/`LIME` helper `_reference_explanation` is cleared on `reset()`.
- [ ] Matplotlib figure count does not grow during repeated headless `render()` calls.
- [ ] Memory stress test (100+ iterations) shows stable RSS after fix.
- [ ] `make test` and `make local-checks-pr` pass.

## 7) PlotSpec default-promotion follow-up (ADR-036 / ADR-037)

Goal: make an explicit, recorded decision about whether to promote PlotSpec from opt-in to default; define or ratify the readiness gate that governs promotion; update visualization defaults only if the gate is satisfied.

Why: ADR-037 §5 defers default-path promotion from v0.11.1 and requires a v0.11.2 follow-up. `docs/api/visualization.md` states that any change to the runtime `kind`-registration policy (currently core-only) also requires this explicit decision point. The decision must be recorded in the release plan so the outcome is traceable.

Primary touchpoints:

- `docs/improvement/adrs/ADR-037-visualization-extension-and-rendering-governance.md` (§5 default-path posture, §6 kind extension policy)
- `docs/improvement/adrs/ADR-036-plot-spec-canonical-contract-and-validation-boundary.md` (readiness-gate criteria)
- `docs/improvement/RELEASE_PLAN_v1.md` (ADR-036/ADR-037 roadmap summary lines)
- `docs/api/visualization.md` (note about runtime kind-registration policy)
- `src/calibrated_explanations/viz/builders.py` and `src/calibrated_explanations/core/calibrated_explainer.py` (call sites that choose legacy vs PlotSpec path)

Step-by-step implementation:

1. Audit PlotSpec readiness.
   1. Run the test suite with PlotSpec path forced as default (`pytest -q -k plotspec`); note any failures or parity gaps.
   2. Check `tests/unit/viz/test_plot_parity_adapter.py` and `tests/unit/viz/test_plot_parity_fixtures.py` for known gaps.
   3. Document findings in a short decision note.
2. Define the readiness gate.
   1. If no gate exists: draft gate criteria (e.g., zero parity failures, full `kind` coverage for core plot kinds, no regressions in `test_plotting.py`).
   2. If a gate exists: verify it is current and enforceable.
   3. Record the gate in ADR-037 §5 (or an amendment note) and in ADR-036.
3. Make the promotion decision.
   1. If gate is met → promote PlotSpec to default; update call sites; update `visualization.md` note.
   2. If gate is not met → record the specific gaps; defer promotion to the earliest milestone where gaps can close; update ADR-037 §5 with an explicit deferral note and target milestone.
4. Decide on runtime kind-extension policy.
   1. If no new ADR/amendment is ready → confirm the current policy (core-only) remains in force; update `visualization.md` to remove the "explicit v0.11.2 follow-up decision" forward reference and replace it with the outcome.
   2. If a new ADR/amendment is ready → reference it in `visualization.md`.
5. Record the decision in RELEASE_PLAN_v1.md.
   1. Update the ADR-036 and ADR-037 roadmap summary lines to reflect the v0.11.2 outcome (gate defined, promotion decision taken or deferred with explicit target).

Verification checklist:

- [ ] PlotSpec readiness findings documented (parity test results noted).
- [ ] Readiness gate criteria written and recorded in ADR-037 §5 (or amendment).
- [ ] Explicit promotion decision recorded: either defaults flipped or deferral milestone stated.
- [ ] `docs/api/visualization.md` runtime kind-registration note updated to reflect the v0.11.2 outcome (no open "follow-up" forward reference remains).
- [ ] ADR-036 and ADR-037 roadmap summary lines in `RELEASE_PLAN_v1.md` updated with v0.11.2 outcome.
- [ ] If promotion occurs: call-site changes land with tests; no legacy-path regressions.
- [ ] `make local-checks-pr` passes.
