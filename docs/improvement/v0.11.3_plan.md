The earlier dependency analysis understated the difference between "can start"
and "can be declared closed." The following tasks should not be closed until
their upstream decision or cleanup has finished:

  documenting methods that are immediately removed or missing docstrings for the
  post-removal surface.

  remove `legacy/plotting.py` via caller migration before the Standard-001

  known.** Both tasks add or replace fail-closed tests. If Task 3 changes marker

  ADR and standard gaps, including the ADR-028/STD-005 warning-to-logging

## 7) Add optional uv support for contributor setup and CI install acceleration

Goal: implement the decided packaging workflow posture: support `uv` as an optional contributor setup and CI install-acceleration path, while keeping `pip install calibrated-explanations` as the canonical user installation command and preserving the existing `pip` + `constraints.txt` release path.

Why: the repository contains `uv.lock`, but CI and contributor documentation currently use `pip`, `requirements.txt`, and `constraints.txt`. The lockfile therefore risks looking authoritative while not being validated. v0.11.3 adopts the narrow Option A outcome: use `uv` to improve contributor onboarding and measure CI dependency-install speed, but do not make `uv` authoritative for end-user installation, release builds, or dependency policy.

Decision options and recommendation:

- **Option A: Adopt `uv` as an optional supported workflow**
  - In favor:
    - Can speed up CI dependency installation where install time dominates.
    - Improves contributor onboarding with faster environment creation.
    - Helps notebook/tutorial users reproduce environments more reliably.
    - Keeps `pip` as the canonical public install path while adding a modern fast path.
  - Most important argument against:
    - Adds a second validated packaging workflow that must be maintained.
- **Option B: Defer `uv` support and keep current `pip` workflows**
  - In favor:
    - Avoids packaging churn during RC readiness work.
    - Preserves the currently validated CI and release path.
    - Allows a later evidence-based decision using measured CI timing data.
  - Most important argument against:
    - Leaves the existing `uv.lock` ambiguity unresolved unless separately documented or removed.
- **Option C: Declare `uv` unsupported and remove/document stale artifacts**
  - In favor:
    - Eliminates misleading signals from an unvalidated lockfile.
    - Keeps install and CI guidance simple.
    - Reduces maintenance burden before v1.0.
  - Most important argument against:
    - Gives up a likely CI/install speed improvement and a useful reproducibility path for contributors.

Recommendation: choose Option A, but narrowly. Treat `uv` as an optional contributor and CI-acceleration workflow, not as the canonical user installation path. Start with a scoped `uv pip install -e .[dev]` validation lane or timing experiment before adopting `uv sync` as authoritative. Keep `pip install calibrated-explanations` as the primary user-facing install command.

Decided outcome for v0.11.3:

- Adopt Option A only.
- Keep `pip install calibrated-explanations` as the primary user-facing install command in README and release docs.
- Keep `python -m pip install -e .[dev] -c constraints.txt` as the canonical fallback contributor workflow.
- Add `uv` as an optional fast-path for contributors using `uv pip install -e .[dev] -c constraints.txt`.
- Do not switch release builds, PyPI validation, docs release builds, or the main test matrix to `uv sync` in this task.
- Treat `uv.lock` as a maintained optional workflow artifact only if it is refreshed to match `pyproject.toml`; otherwise remove it or document it as non-authoritative in the same PR.

## 8) Release-hardening task (v0.11.3): Difficulty-aware reject hardening and paper-aligned singleton reporting

- **Task name:** Reject hardening for v0.11.3: difficulty-aware reject evaluation, experimental strategy containment, paper-aligned singleton error-rate reporting, and validity metadata.
- **Status:** Complete on `difficulty_reject`; `make local-checks-pr` passed on 2026-05-23.
- **Branch scope:** This task now covers the full `difficulty_reject` branch, not only the final singleton-accounting hardening slice. The branch documents the existing difficulty path through Venn-Abers, adds opt-in experimental reject-score normalization, adds provenance checks for unsafe difficulty/novelty estimator fitting, evaluates the resulting behavior through Scenarios 8-11, and then hardens reject metadata against the 2025 conformal-reject paper.
- **Runtime scope:** Keep stable public reject behavior unchanged for `ncf="default"` and `ncf="ensured"`. Add and contain `strategy="experimental.difficulty_normalized"` as a research-only path that normalizes reject nonconformity scores before conformal p-values and prediction sets are computed. Add `strategy="experimental.ambiguity_normalized_novelty_penalized"` as a diagnostic novelty/ambiguity research variant, not as a public NCF promotion.
- **Thresholded regression scope (RT-16):** Thresholded regression reject is pure binary event classification, not interval-width selection. Runtime and evaluation must use one shared event contract: scalar thresholds label event `1` iff `y <= threshold`, interval thresholds label event `1` iff `low < y <= high`, invalid intervals raise `ValidationError`, and per-instance threshold arrays are out of scope for reject. Scenario 3 is reframed as thresholded-regression binary-event validity and reports event coverage, singleton precision/recall/error, empty/singleton/ambiguity counts, novelty/ambiguity/reject rates, and structural violations instead of interval-width, accepted-interval-width, interval-coverage, or MSE headline metrics.
- **Scientific metadata scope:** Harden reject metadata so epsilon is treated as conformal set significance (set miscoverage control), while accepted singleton error is reported as a separate sigma-hat estimate with explicit method, significance, definability, raw value, clamped value, and clamping flag. Preserve `error_rate` only as a backward-compatible deprecated alias for the clamped singleton estimate, not as empirical observed error.
- **Paper alignment:** Add centralized singleton accounting for prediction-set size geometry: empty sets are novelty rejections, singleton sets are accepted predictions, and multi-label sets are ambiguity rejections. Do not claim accepted singleton error equals epsilon. For ordinary reporting, use the plug-in estimate at the same epsilon that produced the prediction sets. If offline ICP training-conditional reporting is added, compute `epsilon_tilde` from `delta` and calibration size, recompute prediction sets at `epsilon_tilde`, and compute counts and sigma from those same `epsilon_tilde` sets.
- **Validity and provenance guardrails:** Metadata must distinguish decision epsilon/confidence from singleton-bound epsilon/confidence; include smoothing and validity-claim fields; default deterministic split-conformal paths to no exact online-smoothed validity claim; mark multiclass as `binary_correctness_proxy` and not paper-aligned K-class conformal coverage; and prevent difficulty/novelty strategies from silently presenting calibration-label or residual leakage as conformally valid. Strict provenance raises on unsafe leakage without cross-fitting, while permissive provenance may continue only as heuristic/no conformal validity claim.
- **Evaluation scope:** Scenarios 8-11 are part of this task. ### Goal

Harden the configuration management system against 10 red-team findings by: (1) repairing two ADR-034 lifecycle violations, (2) making `ExplainerBuilder`/`from_config()` partial wiring explicit through a `UserWarning` limited to the two genuinely-unwired fields (`task`, `parallel_workers`) and updated docstrings, (3) deleting zombie configuration code, (4) adding `CE_DEBUG_TRUST_INVARIANTS` to the governed key namespace, (5) documenting `ExplainerBuilder`/env-var precedence for cache and parallel, (6) resolving the `task` and `parallel_workers` fields in `ExplainerConfig` (wire or remove), (7) promoting `ExplainerBuilder`/`ExplainerConfig` to the root public namespace, and (8) recording intentional design boundaries in ADR-034 §7. **Revised extension (2026-06-03):** Task 10 also centralizes the migrated runtime
consumer lifecycle behind a process-level `ConfigManager` singleton and promotes
the configuration schema tables into a class-owned `ConfigSpec`. This preserves
the existing `ConfigManager.from_sources()` CLI-per-invocation behavior while
allowing top-level process boundaries to initialize configuration once.

### Why this task exists

A red-team analysis of the configuration management system (2026-05-27) identified 10 gaps across three dimensions: runtime correctness (ADR-034 lifecycle violations), user-contract clarity (`ExplainerBuilder` fields not wired by `from_config()`), and architectural coherence (zombie code, ungoverned env keys, undocumented precedence rules). Finding 7 (reject policy) is already owned by Task 9 step 9. The remaining 9 findings are addressed here. Without this task, v0.11.3 could close with users silently misconfiguring explainers through `ExplainerBuilder` (expecting `task` and `parallel_workers` to take effect when they do not), with a dead config file that creates a false impression of a live INI-based config infrastructure, and with no explicit record of what the config system supports vs. what is an intentional design boundary.

**Red-team notes on findings 1 and 6**: The original analysis overstated the wiring gap. Code reading of `wrap_explainer.py` lines 225-237 and 425-442 confirmed that `preprocessor`, `auto_encode`, and `unseen_category_policy` ARE wired by `from_config()` (lines 227-229), and `threshold`/`low_high_percentiles` ARE applied at explain_* time via `kwargs.setdefault()` from the stored `_cfg`. Only `task` and `parallel_workers` are genuinely not wired. Finding 6 was written under the false premise that 6 fields needed wiring; with the corrected count, the implementation choice is: wire 2 fields or remove them as not applicable to `WrapCalibratedExplainer`. Either resolution is v0.11.3 work.

1. **(C) `from_config()` silently ignores `task` and `parallel_workers`.** These are the only two `ExplainerConfig` fields not acted upon by `from_config()`: `task` (the `CalibratedExplainer` auto-detects task from the model, so this field has no applicable wiring target in `WrapCalibratedExplainer`) and `parallel_workers` (a "placeholder (not wired yet)" per the code comment — redundant with `perf_parallel_workers` which is already wired via the perf factory). A user who sets `ExplainerBuilder(model).task("classification").parallel_workers(4)` gets silent no-op behavior. Fix by either (a) wiring both fields to their correct targets, or (b) removing them from `ExplainerConfig` as inapplicable/redundant. Whichever path is chosen, emit a `UserWarning` from `build_config()` for any non-default value set on a removed/unwired field, so callers see it at configuration time, not silently at explain_* time.
2. **(C) `perturbation.py` creates a fresh ConfigManager per-test invocation.** `_get_perturbation_config_manager()` calls `ConfigManager.from_sources()` on every call when `PYTEST_CURRENT_TEST` is set — an ADR-034 §3 lifecycle violation. The v0.11.3 extension supersedes the earlier module-specific singleton fix: perturbation now reads through `get_process_config_manager()`, and pytest resets configuration through `reset_process_config_manager_for_testing()`.
3. **(C) `CE_DEBUG_TRUST_INVARIANTS` bypasses ConfigManager.** `plugins/_trust.py` reads this env var directly via `os.getenv`. Adding it to `_KNOWN_ENV_KEYS` eliminates the risk of a spurious "unknown CE_ variable" warning and brings it under governance visibility in `export_effective()`. Full routing through ConfigManager is not attempted because it would require `_trust.py` to import ConfigManager, risking a circular import via `plugins/registry.py`; the direct `os.getenv` read is documented in ADR-034 §7 as a sanctioned exception.
4. **(C) Zombie `config.ini`.** `src/calibrated_explanations/utils/configurations/config.ini` declares `[perturbation_variables]` and `[figure_size_variables]` sections but nothing reads them — the configparser import in `perturbation.py` is commented out. The file creates a false impression of a live INI-based config infrastructure and should be deleted.
5. **(C) `ExplainerBuilder`/env-var precedence is undocumented.** `CE_CACHE=off` silently overrides `ExplainerBuilder.perf_cache(enabled=True)` because `CacheConfig.from_env()` / `ParallelConfig.from_env()` runs inside `_build_perf_factory()` after builder construction. This interaction is not documented anywhere visible to users. Fix by adding a Notes section to `perf_cache()` and `perf_parallel()` docstrings in `api/config.py`, and recording the rule in ADR-034 §7.
6. **(C) `task` and `parallel_workers` must be resolved before v0.11.3 ships.** As established in finding 1, these are the only genuinely-unwired fields. Resolution: (a) `task` — `WrapCalibratedExplainer` auto-detects task from the model; the `task` field in `ExplainerConfig` has no applicable wiring target in `WrapCalibratedExplainer` and should be removed from `ExplainerConfig`. If any caller sets a non-default `task`, emit a `UserWarning` from `build_config()` saying the field is not applicable and will be removed. (b) `parallel_workers` — this field is explicitly marked "not wired yet" in the code comment and is redundant with `perf_parallel_workers` which is fully wired via the perf factory; remove it from `ExplainerConfig`. Same `UserWarning` pattern. Both removals are safe because `ExplainerConfig`/`ExplainerBuilder` are not exported from the root namespace (addressed in finding 9) and are currently internal/future-facing per the module docstring.
7. **(Task 9 step 9 — excluded from Task 10 scope) Reject policy config integration.** No `CE_REJECT_POLICY` env var, no `[tool.calibrated_explanations.reject]` pyproject.toml section, no `ConfigManager` reads in the reject framework. This finding is already recorded as step 9 in Task 9. Task 10 does not own this; it is excluded to prevent duplicate ownership. Ensure Task 9 step 9 is marked complete before closing Task 9.
8. **(C) Document intentional two-system plot config design in ADR-034 §7.** Plot aesthetic settings (fonts, DPI, colors) are governed by a separate `plot_config.ini` via `configparser` in `plotting.py:load_plot_config()`, with no connection to `ConfigManager`. This is intentional design, not a gap: `ConfigManager` governs behavioral/deployment configuration (plugin selection, trust/deny, telemetry, cache, parallel); `plot_config.ini` governs visual aesthetics. These serve different concerns. Bridging them would add complexity without benefit. Record this design boundary explicitly in ADR-034 §7. No code change needed. (Note: there is no dependency on Task 1 for this finding — `plot_config.ini` is read by `plotting.py`, not `legacy/plotting.py`.)
9. **(C) `ExplainerBuilder` and `ExplainerConfig` are absent from the root namespace.** Neither is in `calibrated_explanations.__all__` or `__getattr__`. A user wanting to use the programmatic config workflow must import from `calibrated_explanations.api.config`, which is not obvious. This is a 5-line additive change: add both to `__init__.py` `__all__` and the lazy `__getattr__` dispatch. Update ADR-020 to record the new exports. `ConfigManager` is intentionally not promoted to the root namespace — it is an infrastructure primitive for advanced use; its import path `calibrated_explanations.core.config_manager` is acceptable.
10. **(C) Document env-only-by-design keys in ADR-034 §7.** `CE_CACHE`, `CE_PARALLEL`, `CE_PARALLEL_MIN_BATCH_SIZE`, `CE_FEATURE_FILTER`, `CE_STRICT_OBSERVABILITY` resolve `(None, None, None)` in `_RESOLUTION_SPEC` — they are intentionally env-or-programmatic-only in v0.11.x with no pyproject.toml mapping. This is a deliberate design choice. Document it as such in ADR-034 §7. No code change needed.

## 13) Normalize guarded explanations as a parameterized explanation policy

### Goal

Replace the canonical guarded explanation API with parameterized factual and alternative calls:
`explain_factual(..., guarded=True)` and `explore_alternatives(..., guarded=True)`.
Deprecate `explain_guarded_factual(...)` and `explore_guarded_alternatives(...)` as
compatibility wrappers. Add `guarded: bool` to the explanation request path and add exactly
one plugin metadata field, `supports_guarded: bool`, to declare guarded compatibility. Do not
add guarded-specific explanation modes or redundant guarded capability tags. Preserve guarded
audit behavior and update ADRs, documentation, and tests.

### Why this task exists

The pre-existing `explain_guarded_factual` / `explore_guarded_alternatives` methods create a
parallel public API surface that contradicts the CE-first design goal: one entry point per
explanation kind, with options as parameters. Having separate method names for guarded execution
forces callers to choose between two parallel surfaces and prevents the plugin contract from
representing guarded capability cleanly. This task eliminates the parallel surface, makes
guarded a policy parameter (consistent with how `reject_policy`, `threshold`, and
`low_high_percentiles` work), and establishes a single plugin metadata field (`supports_guarded`)
as the only guarded contract signal.

### Governing ADRs / standards / release-plan references

- `docs/improvement/adrs/ADR-032-guarded-explanation-semantics.md` (updated in this task to
  reflect the parameterized canonical API).
- `docs/improvement/adrs/ADR-011-deprecation-and-migration-policy.md` (deprecation wrapper
  policy; removal target v1.0.0).
- `docs/improvement/adrs/ADR-020-legacy-user-api-stability.md` (public API change governance).
- `docs/improvement/adrs/ADR-015-explanation-plugin.md` (plugin contract; `supports_guarded`
  metadata field).

### Compatibility posture

- `explain_factual(guarded=False)` and `explore_alternatives(guarded=False)` (default) preserve
  all existing unguarded behavior without change.
- `explain_guarded_factual(...)` and `explore_guarded_alternatives(...)` continue to work but
  emit `DeprecationWarning` and delegate to the parameterized API.
- Callers that catch the guarded audit (`get_guarded_audit()`) are unaffected.
- Plugin metadata that does not include `supports_guarded` defaults to `False`; no existing
  plugin registration breaks.

### Non-goals

- Removing `explain_guarded_factual` / `explore_guarded_alternatives` in this task (removal is
  v1.0.0; this task only deprecates them).
- Adding new guarded explanation modes (`"guarded_factual"`, `"guarded_alternative"`).
- Adding redundant guarded capability tags in plugin capabilities.

### Primary touchpoints

- `src/calibrated_explanations/plugins/explanations.py` — `ExplanationRequest.guarded`
- `src/calibrated_explanations/plugins/base.py` — `validate_plugin_meta` `supports_guarded`
- `src/calibrated_explanations/plugins/builtins.py` — builtin plugin metadata
- `src/calibrated_explanations/plugins/registry.py` — `find_explanation_plugin_for`
- `src/calibrated_explanations/core/calibrated_explainer.py` — `explain_factual`,
