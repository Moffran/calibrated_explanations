### Goal

Remove the three deprecated compatibility surfaces that carry an explicit
v0.11.3 removal horizon in the Standard-001 ledger:
`src/calibrated_explanations/legacy/plotting.py`,
`serialization.validate_payload`, and `viz.builders.legacy_get_fill_color`.
After removal, update the enforcement allowlist so the checker treats
re-introduction as a violation, and close Standard-001 in the remediation
document.

### Why this task exists

ADR-011 § "v1.0 finalization exception" is binding: v1.0.0 must ship with zero surviving deprecations. Standard-001 recorded these three surfaces as bounded compatibility surfaces with a v0.11.3 removal horizon in v0.11.1. The finalization exception mandates their removal in this milestone; there is no deferral path. Standard-001 closure is also a named gate condition for v1.0.0.

### Hardening / red-team findings incorporated (2026-05-12)

1. **`legacy/plotting.py` is not a thin wrapper.** It is 864 lines of substantive plotting logic. It has two active runtime callers outside `legacy/`: `plotting.py` (8+ import sites) and `plugins/builtins.py`. Removal requires caller migration first, not simple deletion.
2. **ADR-024 and ADR-025 are superseded** (2025-10-24). The previous task description's instruction to "consult ADR-024/025" referenced stale docs. The maintenance reference is `docs/maintenance/legacy-plotting-reference.md`.
3. **`serialization.validate_payload` is in `serialization.py`, not `legacy/`.** It is a 6-line wrapper at `serialization.py` lines 193–199, exported in `__all__` at line 202.
4. **Only the public color alias is deprecated.** `legacy_get_fill_color = _legacy_get_fill_color` at `viz/builders.py` line 173 is the removal target. The underlying `_legacy_get_fill_color` (lines 151–169) is used for runtime constants and must not be removed.
5. **`check_std001_nomenclature.py` currently allowlists all three deprecated surfaces.** CI passes with them in place. After each removal, the corresponding allowlist entry must be deleted so re-introduction fails CI.
6. **No fail-closed tests exist for these removals.** Current tests verify backward-compat parity and delegation, which is correct while the deprecated surfaces exist and wrong after removal. Tests must be replaced with attribute-absence assertions or old-module import-failure assertions.
7. **Task 6 is no longer an escape hatch for deleting `legacy/plotting.py`.** Default promotion landed, but explicit opt-out and fallback imports still rely on `legacy/plotting.py`; the module must be removed by caller migration.
8. **The legacy plotting module has script and test dependencies, not just runtime callers.** `scripts/plot_spec/generate_side_by_side.py` and several legacy/plotting parity tests import or patch the old module. Deleting the module without migrating those surfaces would break evidence tooling.

## 7) Add optional uv support for contributor setup and CI install acceleration

Goal: implement the decided packaging workflow posture: support `uv` as an optional contributor setup and CI install-acceleration path, while keeping `pip install calibrated-explanations` as the canonical user installation command and preserving the existing `pip` + `constraints.txt` release path.

Why: the repository contains `uv.lock`, but CI and contributor documentation currently use `pip`, `requirements.txt`, and `constraints.txt`. The lockfile therefore risks looking authoritative while not being validated. v0.11.3 adopts the narrow Option A outcome: use `uv` to improve contributor onboarding and measure CI dependency-install speed, but do not make `uv` authoritative for end-user installation, release builds, or dependency policy.

Decision options and recommendation:

- **Option A: Adopt `uv` as an optional supported workflow**
  - In favor:
    - Can speed up CI dependency installation where install time dominates.
    - Improves contributor onboarding with faster environment creation.
    - Helps notebook/tutorial users reproduce environments more reliably.
    - Keeps `pip` as the canonical public install path while adding a modern fast path.
  - Most important argument against:
    - Adds a second validated packaging workflow that must be maintained.
- **Option B: Defer `uv` support and keep current `pip` workflows**
  - In favor:
    - Avoids packaging churn during RC readiness work.
    - Preserves the currently validated CI and release path.
    - Allows a later evidence-based decision using measured CI timing data.
  - Most important argument against:
    - Leaves the existing `uv.lock` ambiguity unresolved unless separately documented or removed.
- **Option C: Declare `uv` unsupported and remove/document stale artifacts**
  - In favor:
    - Eliminates misleading signals from an unvalidated lockfile.
    - Keeps install and CI guidance simple.
    - Reduces maintenance burden before v1.0.
  - Most important argument against:
    - Gives up a likely CI/install speed improvement and a useful reproducibility path for contributors.

Recommendation: choose Option A, but narrowly. Treat `uv` as an optional contributor and CI-acceleration workflow, not as the canonical user installation path. Start with a scoped `uv pip install -e .[dev]` validation lane or timing experiment before adopting `uv sync` as authoritative. Keep `pip install calibrated-explanations` as the primary user-facing install command.

Decided outcome for v0.11.3:

- Adopt Option A only.
- Keep `pip install calibrated-explanations` as the primary user-facing install command in README and release docs.
- Keep `python -m pip install -e .[dev] -c constraints.txt` as the canonical fallback contributor workflow.
- Add `uv` as an optional fast-path for contributors using `uv pip install -e .[dev] -c constraints.txt`.
- Do not switch release builds, PyPI validation, docs release builds, or the main test matrix to `uv sync` in this task.
- Treat `uv.lock` as a maintained optional workflow artifact only if it is refreshed to match `pyproject.toml`; otherwise remove it or document it as non-authoritative in the same PR.

## 8) Release-hardening task (v0.11.3): Difficulty-aware reject hardening and paper-aligned singleton reporting

- **Task name:** Reject hardening for v0.11.3: difficulty-aware reject evaluation, experimental strategy containment, paper-aligned singleton error-rate reporting, and validity metadata.
- **Status:** In progress on `difficulty_reject`.
- **Branch scope:** This task now covers the full `difficulty_reject` branch, not only the final singleton-accounting hardening slice. The branch documents the existing difficulty path through Venn-Abers, adds opt-in experimental reject-score normalization, adds provenance checks for unsafe difficulty/novelty estimator fitting, evaluates the resulting behavior through Scenarios 8-11, and then hardens reject metadata against the 2025 conformal-reject paper.
- **Runtime scope:** Keep stable public reject behavior unchanged for `ncf="default"` and `ncf="ensured"`. Add and contain `strategy="experimental.difficulty_normalized"` as a research-only path that normalizes reject nonconformity scores before conformal p-values and prediction sets are computed. Add `strategy="experimental.ambiguity_normalized_novelty_penalized"` as a diagnostic novelty/ambiguity research variant, not as a public NCF promotion.
- **Scientific metadata scope:** Harden reject metadata so epsilon is treated as conformal set significance (set miscoverage control), while accepted singleton error is reported as a separate sigma-hat estimate with explicit method, significance, definability, raw value, clamped value, and clamping flag. Preserve `error_rate` only as a backward-compatible deprecated alias for the clamped singleton estimate, not as empirical observed error.
- **Paper alignment:** Add centralized singleton accounting for prediction-set size geometry: empty sets are novelty rejections, singleton sets are accepted predictions, and multi-label sets are ambiguity rejections. Do not claim accepted singleton error equals epsilon. For ordinary reporting, use the plug-in estimate at the same epsilon that produced the prediction sets. If offline ICP training-conditional reporting is added, compute `epsilon_tilde` from `delta` and calibration size, recompute prediction sets at `epsilon_tilde`, and compute counts and sigma from those same `epsilon_tilde` sets.
- **Validity and provenance guardrails:** Metadata must distinguish decision epsilon/confidence from singleton-bound epsilon/confidence; include smoothing and validity-claim fields; default deterministic split-conformal paths to no exact online-smoothed validity claim; mark multiclass as `binary_correctness_proxy` and not paper-aligned K-class conformal coverage; and prevent difficulty/novelty strategies from silently presenting calibration-label or residual leakage as conformally valid. Strict provenance raises on unsafe leakage without cross-fitting, while permissive provenance may continue only as heuristic/no conformal validity claim.
- **Evaluation scope:** Scenarios 8-11 are part of this task. Scenario 8 measures the existing indirect Venn-Abers difficulty effect; Scenario 9 evaluates direct experimental difficulty-normalized reject scoring; Scenario 10 evaluates the ambiguity-normalized novelty-penalized research variant; Scenario 11 performs matched reject-rate operating-point selection. Scenario artifacts and summaries must report epsilon as set significance, separate empirical diagnostics from sigma estimates, expose singleton/empty/ambiguity counts, and avoid treating multiclass proxy metrics as paper-aligned K-class coverage.
- **Documentation scope:** Update `docs/citing.md`, practitioner/researcher reject docs, `evaluation/reject/README.md`, scenario markdown summaries, `docs/improvement/difficulty_normalized_reject_plan.md`, `docs/improvement/difficulty_reject_evaluation_report.md`, `CHANGELOG.md`, and release-plan task text so they state the branch outcome: difficulty-normalized and novelty-aware reject remain experimental, public promotion is not justified by Scenario 11, and paper-aligned singleton reporting is metadata/accounting hardening rather than a new empirical guarantee.

Scenario 11 showed insufficient and unstable accepted-accuracy benefit at matched operating points.
