# v0.11.3 Release Task Implementation Plan

This document is the execution plan for all `v0.11.3 (RC readiness: Standard closure, ADR-030 ratification, docs gap)` tasks listed in `docs/improvement/RELEASE_PLAN_v1.md`.

## Planning standard used in this file

Each task contains:

1. Why the task exists (ADR/standard alignment).
2. Concrete code and doc touchpoints.
3. Ordered implementation steps with dependency order.
4. Required tests/checks and release artifacts.
5. A detailed verification checklist that can be executed as-is.

## Global execution constraints

- Keep CE-first and ADR-first behavior.
- Preserve public API compatibility — shim removals must retain or raise for removed paths per ADR-011.
- Any behavior change must include docs + tests in same PR.
- Validation path for each task PR:
  1. `make local-checks-pr`
  2. Task-targeted pytest runs
  3. `make local-checks` if task changes branch-gate behavior

## 1) Close Standard-001: remove remaining transitional shims

Goal: remove all remaining transitional naming shims from `legacy/` and confirm naming/tooling enforcement is green, completing the Standard-001 remediation plan.

Why: v0.11.1 delivered double-underscore mutation cleanup and confined transitional shims to `legacy/`. This task performs the final removal phase — deleting the shims themselves — so Standard-001 is fully closed before the RC freeze. Standard-001 closure is a required precondition for v1.0.0 (`"Final shim removals completed"`).

Prerequisite: v0.11.1 Standard-001 task (item 8) must be complete and green.

Primary touchpoints:

- `src/calibrated_explanations/legacy/` (all shim modules)
- `docs/improvement/Standard-001_nomenclature_remediation.md`
- Naming lint configuration (Ruff rules, pre-commit config)
- `tests/` — any test that imports from removed shim paths

Step-by-step implementation:

1. Inventory remaining shims.
   1. List all modules in `src/calibrated_explanations/legacy/` that exist solely as transitional re-export shims.
   2. Distinguish shims (thin wrappers that only re-export canonical locations) from maintained legacy implementations (e.g., `legacy/plotting.py` which is an ADR-024/025 maintenance reference — do NOT remove).
   3. Identify any external or internal imports of each shim.
2. Confirm ADR-011 compliance before removal.
   1. Verify each shim to be removed has been emitting `DeprecationWarning` via `deprecate()` for at least two prior minor releases (per ADR-011 two-release window).
   2. If any shim has not been through the deprecation window, do NOT remove it yet — escalate to maintainer and defer.
3. Remove compliant shims.
   1. Delete shim files that have completed their deprecation window.
   2. Update `legacy/__init__.py` to remove references to deleted shims.
   3. Remove or update any re-export entries in `src/calibrated_explanations/__init__.py` or `src/calibrated_explanations/legacy/__init__.py`.
4. Update tests.
   1. Remove or update test imports that relied on removed shim paths.
   2. Add a regression test that confirms removed shim paths raise `ImportError` (not silently succeed).
5. Update docs and enforcement.
   1. Update `docs/improvement/Standard-001_nomenclature_remediation.md` to mark the final shim removal phase complete.
   2. Verify naming lint runs clean on the release branch (`ruff check src/`).

Verification checklist:

- [ ] Only ADR-011-compliant shims (those past their deprecation window) are removed.
- [ ] Maintained legacy implementations (e.g., `legacy/plotting.py`) are NOT removed.
- [ ] Removed shim paths raise `ImportError` — regression test confirms this.
- [ ] No internal runtime code imports removed shim paths.
- [ ] `docs/improvement/Standard-001_nomenclature_remediation.md` updated to reflect final completion.
- [ ] Naming lint (`ruff check src/`) reports zero violations.
- [ ] `make local-checks-pr` passes.

## 2) Close Standard-002 known gap: WrapCalibratedExplainer numpydoc blocks

Goal: add full numpydoc-format docstrings to `WrapCalibratedExplainer` and remaining stable public surfaces identified in the Standard-002 appendix gap, closing the last known Standard-002 gap before the RC freeze.

Why: The RELEASE_PLAN_v1.md appendix identifies one remaining Standard-002 gap: "Wrapper public APIs lack full numpydoc blocks (Severity 12)". Closing this gap in v0.11.3 means v1.0.0-rc only needs to verify the coverage bar is maintained — not implement anything new. Standard-002 compliance at ≥90% is a v1.0.0-rc gate condition.

Primary touchpoints:

- `src/calibrated_explanations/core/wrap_explainer.py`
- Any other public surfaces identified by `pydocstyle` / `ruff` docstring checks
- `docs/improvement/code_documentation_uplift.md`
- `docs/standards/STD-002-code-documentation-standard.md`

Step-by-step implementation:

1. Identify the gap surface.
   1. Run `pydocstyle src/calibrated_explanations/core/wrap_explainer.py` and capture all missing or incomplete docstring violations.
   2. Run the project's docstring coverage tool (as configured in CI) and identify which public methods/classes fall below the threshold.
2. Write numpydoc docstrings.
   1. For each `WrapCalibratedExplainer` public method lacking a full numpydoc block, add:
      - One-line summary.
      - `Parameters` section (all parameters with types and descriptions).
      - `Returns` section (return type and description, or `None`).
      - `Raises` section (any documented exceptions).
      - `Examples` section (at minimum a one-line usage example).
   2. Apply the same to any other public-surface methods identified in step 1.
3. Verify coverage.
   1. Re-run the docstring coverage check to confirm the threshold is met.
   2. Re-run `pydocstyle` to confirm zero violations on touched files.
4. Update Standard-002 appendix.
   1. Mark the "Wrapper public APIs lack full numpydoc blocks" gap as resolved in the RELEASE_PLAN_v1.md appendix.
   2. Add a compliance verification entry for Standard-002 with the current date.

Verification checklist:

- [ ] All `WrapCalibratedExplainer` public methods have full numpydoc blocks (Summary, Parameters, Returns, Raises, Examples).
- [ ] `pydocstyle` reports zero violations on `wrap_explainer.py` and other touched files.
- [ ] Docstring coverage check passes at ≥90% threshold.
- [ ] Standard-002 appendix gap marked resolved with current date.
- [ ] `make local-checks-pr` passes.

## 3) Ratify ADR-030 zero-tolerance enforcement

Goal: confirm the ADR-030 zero-tolerance enforcement posture is CI-blocking, document marker hygiene rules formally in the ADR, and declare mutation testing as optional for core modules — completing the ADR-030 ratification before GA.

Why: v0.11.0 delivered the tooling extension (assertion and determinism checks in `detect_test_anti_patterns.py`). v1.0.0 originally held the ratification task, but ratification should inform RC readiness, not be deferred to GA. Completing it in v0.11.3 ensures the RC freeze is against a fully ratified enforcement posture.

Primary touchpoints:

- `docs/improvement/adrs/ADR-030-test-quality-priorities-and-enforcement.md`
- `.github/workflows/` (CI quality gate jobs)
- `scripts/quality/detect_test_anti_patterns.py`
- `scripts/local_checks.py`

Step-by-step implementation:

1. Verify CI enforcement is blocking.
   1. Confirm `detect_test_anti_patterns.py --check` is wired into CI as a blocking gate (not advisory).
   2. Confirm the assertion-check and determinism-check extensions added in v0.11.0 are included in the blocking run.
   3. If any check is advisory-only, promote it to blocking and update the CI workflow accordingly.
2. Document marker hygiene rules in ADR-030.
   1. Add or update a section in ADR-030 defining the required pytest marker taxonomy: `unit`, `integration`, `slow`, and any project-specific markers.
   2. Document the rule that every test must carry at least one of the canonical markers.
   3. Document how to add markers and where the enforced marker list is maintained.
3. Declare mutation testing policy.
   1. Add a section in ADR-030 stating that mutation testing is optional for core modules: recommended for critical-path logic, not required for release gates.
   2. Note which tool is suggested (e.g., `mutmut`) and how to run it locally.
4. Update ADR-030 status note and ratification record.
   1. Add a ratification note to the ADR status note field: "Ratified v0.11.3 — zero-tolerance CI enforcement confirmed, marker hygiene documented, mutation testing policy declared."
   2. Update the status note date.

Verification checklist:

- [ ] `detect_test_anti_patterns.py --check` is a CI-blocking gate (not advisory).
- [ ] Assertion-check and determinism-check extensions are included in the blocking run.
- [ ] ADR-030 contains a marker hygiene section with canonical marker taxonomy.
- [ ] ADR-030 contains a mutation testing section declaring it optional.
- [ ] ADR-030 status note includes ratification record with date.
- [ ] `make local-checks-pr` passes.

## 4) (Stretch) OSS performance harness template

Goal: provide a reusable, repository-resident harness for semi/fully-online latency measurements with README-level guidance and a reproducible sample run, so adopters have a documented starting point for CE performance profiling.

Why: Calibrated explanations can be resource-intensive at large batch sizes. A documented harness lowers the barrier for contributors and integrators to profile performance, supports the caching/parallel validation that v1.0.0-rc must sign off on, and was deferred from v1.0.0-rc as a self-contained item. This task is stretch — if capacity is constrained, it may be deferred to v1.0.0-rc without blocking the release gate.

Primary touchpoints:

- `scripts/perf/` (existing performance scripts directory)
- New: `scripts/perf/online_latency_harness.py`
- New: `docs/contributor/performance_harness.md`
- `README.md` (or contributor docs cross-link)

Step-by-step implementation:

1. Define harness scope.
   1. Determine the representative workload: batch sizes (e.g., 100, 1000, 5000 rows), explanation modes (factual, alternative), and calibration modes (classification, regression).
   2. Define the latency metrics to capture: wall-clock time per batch, p50/p95 latency, and peak memory (via `tracemalloc` or `psutil`).
2. Implement `online_latency_harness.py`.
   1. Accept CLI arguments for dataset size, explanation mode, number of repetitions, and output format (CSV or JSON).
   2. Use `WrapCalibratedExplainer` CE-first pattern throughout.
   3. Emit a machine-readable results file with fields: `mode`, `n_samples`, `n_features`, `elapsed_seconds`, `peak_memory_mb`, `ce_version`, `timestamp`.
   4. Include a short `--help` output explaining the parameters.
3. Add `docs/contributor/performance_harness.md`.
   1. Document prerequisites (dataset, extra dependencies if any).
   2. Show a sample invocation and expected output range.
   3. Explain how to interpret results and where to report regressions.
4. Commit a sample run artifact.
   1. Run the harness on a representative toy dataset.
   2. Commit the results file to `reports/perf/harness_sample_run.json` as a reference baseline.

Verification checklist:

- [ ] `scripts/perf/online_latency_harness.py` is runnable with `python scripts/perf/online_latency_harness.py --help`.
- [ ] Harness emits machine-readable results file with required fields.
- [ ] `docs/contributor/performance_harness.md` documents invocation and interpretation.
- [ ] Sample run artifact committed to `reports/perf/`.
- [ ] `make local-checks-pr` passes (harness is not wired into blocking CI; it is a local tool).
